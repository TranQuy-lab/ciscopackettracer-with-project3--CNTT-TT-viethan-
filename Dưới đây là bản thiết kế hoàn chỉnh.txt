BẢN THIẾT KẾ HOÀN CHỈNH ĐÃ CHỈNH THEO MÔ HÌNH CORE → SW-TRANSIT-ASA → ASA VÀ LIGHTWEIGHT AP TOÀN CAMPUS
Bản này được chỉnh lại theo phần đầu bạn đã sửa: thay vì để ASA nối trực tiếp vào một Core duy nhất, mô hình dùng thêm SW-TRANSIT-ASA làm switch trung gian Layer 2 cho VLAN 100. Nhờ vậy cả `CORE-SW1` và `CORE-SW2` đều có đường vật lý ra ASA. Khi `CORE-SW1` lỗi, `CORE-SW2` vẫn có thể đi qua `SW-TRANSIT-ASA` để ra ASA.

Điều chỉnh mới: tất cả các phòng ban/khu vực người dùng đều được bố trí Lightweight Access Point (LAP). Mỗi AP chỉ dùng VLAN quản lý AP để lấy IP và join về WLC trung tâm. Hai SSID `WIFI\_GIAOVIEN` và `WIFI\_SINHVIEN` được cấu hình tập trung trên WLC, sau đó được phát trên toàn bộ AP của các phòng ban. Người dùng không dây sẽ vào VLAN 60 hoặc VLAN 70 theo SSID, không cấu hình VLAN người dùng trực tiếp trên AP.
---
1. Phân tích kiến trúc tổng thể
1.1. Mô hình đề xuất
Mạng campus chia thành 2 site:
Zone 1 — Khu K / Campus trung tâm
```text
                             INTERNET / ISP
                                  |
                              EDGE-K Router
                                  |
                           ASA Firewall
                                  |
                           SW-TRANSIT-ASA
                            /           \\
                           /             \\
                    CORE-SW1 ========== CORE-SW2
                       |                   |
                       |                   |
              Access Switches       Access Switches
              Server / PC / AP      Server / PC / AP / WLC
```
Ý nghĩa chỉnh sửa:
`CORE-SW1` và `CORE-SW2` vẫn là 2 Core Layer 3 chạy HSRP, RSTP, routing VLAN, ACL.
`SW-TRANSIT-ASA` là switch Layer 2 trung gian, chỉ làm nhiệm vụ gom kết nối VLAN 100 giữa 2 Core và ASA.
ASA không cắm trực tiếp vào một Core duy nhất nữa.
ASA inside dùng IP `10.10.100.4/29`.
Hai Core trỏ default route về ASA inside `10.10.100.4`.
ASA trỏ route ngược về LAN thông qua HSRP VIP `10.10.100.1`.
Zone 2 — Khu A / Remote Site
```text
Internet / ISP
     |
 EDGE-A Router
     |
 AS-KHUA Switch
     |
 AP Khu A / Wireless Clients
```
Khu K và Khu A kết nối với nhau bằng IPsec Site-to-Site VPN giữa `EDGE-K` và `EDGE-A`.

1.3. Giới hạn trong Cisco Packet Tracer
Công nghệ	Thực tế	Packet Tracer
VSS / StackWise Virtual / vPC	Cho phép EtherChannel từ Access lên 2 Core khác nhau	Không hỗ trợ đầy đủ
Multi-Chassis EtherChannel	Gom link lên 2 switch vật lý độc lập	Không mô phỏng đúng
ASA nâng cao	Có NAT, VPN, firewall policy đầy đủ	Có thể giới hạn tùy phiên bản
WLC FlexConnect	AP remote local switching qua VPN	Packet Tracer hỗ trợ hạn chế
QoS nâng cao	Có shaping, policing, CBWFQ	Packet Tracer hỗ trợ hạn chế
Vì vậy, trong Packet Tracer:
Core và Distribution có thể gộp chung thành `CORE-SW1` và `CORE-SW2`.
Access Switch dual-home lên cả 2 Core bằng 2 trunk riêng.
Không cấu hình EtherChannel từ một Access Switch lên đồng thời 2 Core độc lập.
EtherChannel dùng chắc chắn nhất giữa `CORE-SW1` và `CORE-SW2`.
`SW-TRANSIT-ASA` dùng để giải quyết bài toán cả 2 Core đều có đường vật lý ra ASA.
AP Khu A join WLC qua VPN có thể mô phỏng bằng routing + WLC IP thủ công hoặc DHCP Option 43.

2.5. Kế hoạch triển khai Lightweight AP cho tất cả phòng ban
Nguyên tắc mới của hệ thống WiFi:
Tất cả phòng ban/khu vực người dùng đều có ít nhất 1 Lightweight AP.
AP không tự cấu hình SSID độc lập, mà được WLC trung tâm quản lý.
Port nối AP trên switch để access VLAN 80 tại Campus Khu K.
AP chỉ cần IP quản lý trong VLAN 80 để join WLC `10.10.99.30`.
SSID `WIFI\_GIAOVIEN` được WLC map vào VLAN 60.
SSID `WIFI\_SINHVIEN` được WLC map vào VLAN 70.
Toàn bộ AP trong các phòng ban đều phát cả 2 SSID: `WIFI\_GIAOVIEN` và `WIFI\_SINHVIEN`.

3. Sơ đồ kết nối logic bằng text
```text
                             INTERNET / ISP
                                  |
                         +--------+--------+
                         |                 |
                      EDGE-K            EDGE-A
                         |                 |
                    ASA Firewall       AS-KHUA
                         |                 |
                   SW-TRANSIT-ASA      AP Khu A
                    /           \\
                   /             \\
              CORE-SW1 ======= CORE-SW2
              |    |    |       |    |    |
              |    |    |       |    |    |
         AS-SERVER AS-BGH AS-ESTI AS-LAB-B2 AS-ABC-AP AS-D1
             |       |      |        |         |        |
          Servers PC+LAP PC+LAP  PC+LAP   LAP A/B/C PC+LAP
                                                |
                                      WLC trung tâm quản lý toàn bộ LAP
```
Ý nghĩa:
`CORE-SW1` và `CORE-SW2` kết nối bằng EtherChannel LACP.
Access Switch nối trunk lên cả 2 Core.
`SW-TRANSIT-ASA` gom VLAN 100 từ 2 Core xuống ASA.
ASA không cần biết Core nào đang Active; ASA chỉ route về HSRP VIP `10.10.100.1`.
Nếu Core 1 chết, Core 2 vẫn còn đường vật lý đến ASA thông qua `SW-TRANSIT-ASA`.
WLC đặt tại Khu K, kết nối trunk vào Core.
AP Khu A join WLC qua VPN.

5.2. STP
Nguyên tắc:
VLAN nào HSRP Active trên Core nào thì Core đó là STP Root Primary.
Core còn lại là Root Secondary.
`SW-TRANSIT-ASA` không đặt root, chỉ là switch trung gian.
VLAN	HSRP Active	STP Root Primary
10,20,55,60,80,99,100	CORE-SW1	CORE-SW1
30,40,50,70	CORE-SW2	CORE-SW2
Dùng:
```bash
spanning-tree mode rapid-pvst
```
---
5.3. OSPF
Trong mô hình Packet Tracer:
OSPF chạy nội bộ trên Core nếu cần mở rộng mô hình.
Core quảng bá các mạng VLAN nội bộ.
ASA nên dùng static route để đơn giản và ổn định trong Packet Tracer.
Core có default route về ASA.
ASA có route ngược về `10.10.0.0/16` qua HSRP VIP `10.10.100.1`.
Routing chính:
```text
CORE-SW1/2 default route → ASA inside 10.10.100.4
ASA default route → EDGE-K 172.16.0.2
EDGE-K default route → ISP 203.0.113.1
```
---
5.4. NAT/PAT
NAT thực hiện trên Edge Router:
`EDGE-K` NAT overload cho `10.10.0.0/16`.
`EDGE-A` NAT overload cho `10.20.0.0/16`.
Traffic VPN giữa `10.10.0.0/16` và `10.20.0.0/16` không bị NAT.
Logic NAT exemption:
```text
Nếu source 10.10.0.0/16 đi destination 10.20.0.0/16 → không NAT
Nếu source 10.10.0.0/16 đi Internet → NAT overload
```
---
5.5. VPN Site-to-Site
VPN IPsec giữa:
Site	Router	Public IP
Khu K	EDGE-K	203.0.113.2
Khu A	EDGE-A	198.51.100.2
Traffic được mã hóa:
```text
10.10.0.0/16 <==== IPsec VPN ====> 10.20.0.0/16
```
VPN dùng cho:
Người dùng Khu A truy cập tài nguyên Khu K.
AP Khu A gửi CAPWAP về WLC Khu K.
Quản trị thiết bị Khu A từ Khu K.
---
5.6. Wireless
WLC đặt tại Khu K và đóng vai trò trung tâm điều khiển toàn bộ Lightweight AP.
Interface WLC	VLAN	IP WLC	Gateway
Management	99	10.10.99.30/24	10.10.99.1
Dynamic WIFI_GIAOVIEN	60	10.10.60.10/24	10.10.60.1
Dynamic WIFI_SINHVIEN	70	10.10.64.10/21	10.10.64.1
SSID mapping trên WLC:
SSID	VLAN người dùng	Đối tượng	Phạm vi phát sóng
WIFI_GIAOVIEN	60	Giảng viên / nhân viên	Tất cả Lightweight AP trong phòng ban
WIFI_SINHVIEN	70	Sinh viên	Tất cả Lightweight AP trong phòng ban
AP Management:
AP khu vực	VLAN Management	Gateway	Cách join WLC
AP Khu K	VLAN 80	10.10.80.1	DHCP Option 43 hoặc cấu hình manual WLC IP `10.10.99.30`
AP Khu A	VLAN 280	10.20.80.1	Join WLC qua VPN Site-to-Site
Nguyên tắc cấu hình AP sau điều chỉnh:
Tất cả AP trong các phòng ban là Lightweight AP, không phải Autonomous AP.
AP không tự cấu hình SSID cục bộ; SSID được tạo trên WLC.
Port switch nối AP tại Khu K để `switchport mode access` và `switchport access vlan 80`.
Port switch nối AP tại Khu A để `switchport mode access` và `switchport access vlan 280`.
Trunk từ Core tới WLC phải mang VLAN 60,70,99. Có thể allow thêm VLAN 80 nếu muốn quản trị/mô phỏng thuận tiện, nhưng VLAN 80 không phải VLAN người dùng WiFi.
Toàn bộ AP đều phát cả 2 SSID: `WIFI\_GIAOVIEN` và `WIFI\_SINHVIEN`.
AP join WLC bằng một trong 3 cách:
DHCP Option 43 trỏ về `10.10.99.30`.
DNS name: `cisco-capwap-controller` trỏ về `10.10.99.30`.
Cấu hình manual WLC IP trên AP.
Kết luận phần WiFi: người dùng chọn SSID nào thì WLC đưa họ vào VLAN tương ứng. AP chỉ là thiết bị phát sóng và tạo đường hầm CAPWAP về WLC, không phải nơi quyết định VLAN người dùng.


Chọn `restrict` thay vì `shutdown` để dễ mô phỏng và troubleshooting. Trong mạng thật, port quan trọng có thể dùng `shutdown`.

```bash
switchport mode access

shutdown
```

5.9. QoS
Yêu cầu: `WIFI\_SINHVIEN` được băng thông lớn nhất nhưng không làm nghẽn hoàn toàn các mạng khác.
Đề xuất tại Edge Router:
Class	Source	Bandwidth
WIFI_SINHVIEN	10.10.64.0/21	50%
WIFI_GIAOVIEN	10.10.60.0/24	20%
STAFF / SERVER / khác	Các VLAN còn lại	30%
Logic:
Sinh viên dùng WiFi đông nhất nên cấp nhiều băng thông.
Giảng viên vẫn có băng thông tối thiểu.
Traffic quản trị, server, hành chính không bị starvation.

9.5. Lỗi 3: HSRP sai Active/Standby
Kiểm tra:
```bash
show standby brief
```
Nguyên nhân:
Priority sai.
Chưa cấu hình `preempt`.
VLAN interface down.
Trunk không mang VLAN đó.
Cách sửa mẫu:
```bash
interface vlan40
 standby 40 priority 110
 standby 40 preempt
```

10. Kết luận thiết kế
Thiết kế sau khi chỉnh đáp ứng các yêu cầu chính:
Có mô hình Core — Distribution — Access, trong Packet Tracer gộp Core/Distribution hợp lý.
Có 2 Core Layer 3 chạy HSRP chia tải gateway.
Có `SW-TRANSIT-ASA` để cả 2 Core đều có đường vật lý ra ASA.
Không còn lỗi thiết kế kiểu chỉ một Core có dây đi ASA.
STP Root đi cùng hướng với HSRP Active.
Có EtherChannel LACP giữa 2 Core.
Access Layer dùng trunk riêng lên 2 Core, không dùng Multi-Chassis EtherChannel vì Packet Tracer không hỗ trợ VSS/vPC.
Có phân VLAN rõ ràng cho Server Farm, BGH, Staff, Student, ESTI, WiFi, AP Management, Network Management và Core-ASA Transit.
Có kế hoạch Server Farm rõ ràng: Web Server, DHCP/DNS Server, Database Server, File/App Server.
Có ACL chặn sinh viên truy cập vùng nhạy cảm nhưng vẫn có thể cho truy cập Web Server nội bộ.
Có ASA Firewall, Edge Router, NAT Overload, Site-to-Site IPsec VPN.
Có WLC quản lý AP tập trung, SSID tách VLAN.
Có phương án AP Khu A join WLC qua VPN.
Có Port Security, PortFast, BPDU Guard.
Có QoS logic ưu tiên băng thông lớn nhất cho WIFI_SINHVIEN nhưng vẫn giữ băng thông tối thiểu cho mạng khác.
Mô hình cuối cùng nên dùng:
```text
CORE-SW1
   \\
    SW-TRANSIT-ASA ---- ASA ---- EDGE-K ---- ISP
   /
CORE-SW2
```
Đây là mô hình hợp lý hơn cho Packet Tracer vì vừa giữ được HSRP trên 2 Core, vừa đảm bảo khi một Core lỗi thì Core còn lại vẫn còn đường vật lý để đi ra Firewall.
